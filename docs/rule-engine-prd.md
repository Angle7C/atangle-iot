# 规则引擎 PRD
版本：V1.0  
文档类型：产品需求文档（PRD）  
适用对象：产品经理、设计师、后端开发、前端开发、测试人员

## 1. 产品背景
在 IoT 平台中，设备会持续上报属性、事件和在线状态，同时平台也需要基于业务场景自动触发通知、记录、控制和联动操作。当前平台已有设备、产品、属性、事件、服务等基础能力，但缺少一套可配置、可扩展、可执行的规则引擎能力，用于承接自动化判断与自动化动作。

本期规则引擎以“条件节点 + 动作节点”为核心建模方式，支持平台用户通过可视化或表单化配置，将设备数据变化、设备事件、状态变化转化为可执行的业务联动流程。

## 2. 产品目标
- 提供统一的规则引擎产品能力，支持设备场景下的自动化判断与自动化执行。
- 将规则配置抽象为“条件节点 + 动作节点”，降低理解门槛和使用门槛。
- 支持基于设备属性、设备事件、设备状态进行条件判断。
- 支持执行通知、日志记录、标签处理、分组调整、设备服务调用等动作。
- 为后续告警中心、自动运营、批量任务和联动控制提供统一规则底座。

## 3. 目标用户

### 3.1 平台运营人员
- 配置日常运营规则。
- 为设备自动打标签或调整分组。
- 配置异常通知和自动处理流程。

### 3.2 运维人员
- 配置设备异常联动规则。
- 在设备离线、属性异常、事件触发时执行通知或记录动作。

### 3.3 业务管理员
- 按业务需求配置专题规则。
- 让设备数据变化自动触发业务处理动作。

## 5. 核心问题与价值
- 设备数据变化频繁，纯人工监控成本高。
- 设备异常、业务事件、状态变化缺少自动化处理手段。
- 平台已有设备数据基础，但未形成“检测 -> 判断 -> 执行”的自动化闭环。
- 规则引擎可提升异常响应速度、减少人工操作、增强平台智能化程度。

## 6. 产品定位
规则引擎是设备平台中的自动化能力中心，其职责不是替代产品管理和设备管理，而是将已有设备数据、产品能力和设备操作串联成可配置规则。

规则引擎在本期仅聚焦两类节点：
- 条件节点：定义“何时触发”
- 动作节点：定义“触发后做什么”

## 8. 规则模型定义

### 8.1 规则对象
规则是平台中的自动化配置单元，由基础信息、触发范围、条件节点、动作节点和执行状态组成。

建议包含的业务字段：
- 规则名称
- 规则编码
- 规则说明
- 所属业务场景
- 适用设备范围
- 启用状态
- 条件节点列表
- 动作节点列表
- 创建人
- 创建时间
- 更新时间

### 8.2 条件节点
条件节点用于定义规则生效的判断条件，回答“何时触发规则”。

条件节点由以下信息构成：
- 条件类型
- 数据来源
- 比较字段
- 比较运算符
- 比较值
- 时间窗口
- 条件组合关系

### 8.3 动作节点
动作节点用于定义条件满足后的执行动作，回答“触发后执行什么”。

动作节点由以下信息构成：
- 动作类型
- 执行目标
- 动作参数
- 执行顺序
- 执行失败策略

## 9. 功能需求

### 9.1 规则列表
功能目标：帮助用户统一管理所有规则。

#### 9.1.1 展示字段
- 规则名称
- 规则编码
- 业务场景
- 适用范围
- 启用状态
- 最近执行时间
- 最近执行结果
- 创建人
- 更新时间

#### 9.1.2 筛选条件
- 关键字：规则名称、规则编码
- 启用状态
- 业务场景
- 创建时间范围

#### 9.1.3 支持操作
- 新建规则
- 编辑规则
- 查看详情
- 启用规则
- 停用规则
- 删除规则
- 查看执行日志
- 测试规则

#### 9.1.4 交互规则
- 默认按更新时间倒序。
- 启用状态需在列表中可快速切换。
- 删除规则需二次确认。
- 处于启用状态的规则删除前需先停用或提示风险。

### 9.2 规则基础信息配置
功能目标：定义规则的基本身份和适用范围。

#### 9.2.1 字段要求
- 规则名称：必填，建议 1 到 100 字。
- 规则编码：必填，全局唯一。
- 规则说明：非必填。
- 业务场景：必填，例如设备告警、自动打标、设备控制、通知联动。
- 适用范围：必填，用于定义规则作用对象。

#### 9.2.2 适用范围支持方式
- 按产品范围
- 按设备分组范围
- 按设备标签范围
- 按指定设备范围

#### 9.2.3 范围规则
- 一条规则至少需要指定一种适用范围。
- 多个范围条件之间默认按“且”处理或按产品定义统一收口，具体以前端交互口径为准。
- 若范围内设备为空，规则可以保存但执行时不生效。

### 9.3 条件节点配置
功能目标：定义规则触发判断逻辑。

#### 9.3.1 支持的条件类型
- 设备属性条件
- 设备事件条件
- 设备状态条件
- 时间条件

#### 9.3.2 设备属性条件
用于基于设备属性值触发规则。

配置项：
- 数据源：设备属性
- 属性标识
- 运算符
- 比较值
- 时间窗口

支持运算符：
- 等于
- 不等于
- 大于
- 大于等于
- 小于
- 小于等于
- 包含
- 不包含
- 为空
- 不为空

示例：
- 温度 > 40
- 电量 <= 20
- 运行模式 = 自动

#### 9.3.3 设备事件条件
用于基于设备事件上报触发规则。

配置项：
- 数据源：设备事件
- 事件编码
- 事件类型
- 事件内容匹配条件
- 时间窗口

示例：
- 上报“overheat”事件时触发
- 故障类型为“sensor_error”时触发

#### 9.3.4 设备状态条件
用于基于设备业务状态或在线状态触发规则。

配置项：
- 状态类型：业务状态或在线状态
- 状态值
- 持续时长

示例：
- 在线状态 = 离线 且持续 10 分钟
- 业务状态 = 停用

#### 9.3.5 时间条件
用于在指定时间范围内才允许规则执行。

配置项：
- 生效时间段
- 生效日期范围
- 星期范围

示例：
- 仅工作日 09:00 到 18:00 生效
- 仅夜间值守时段触发

#### 9.3.6 条件组合规则
- 支持多个条件节点组合。
- 条件组合关系支持“且”和“或”。
- 本期默认支持单层组合，不支持多层嵌套条件树。
- 若未配置条件节点，则规则不可保存。

#### 9.3.7 条件校验规则
- 比较值类型必须与属性值类型匹配。
- 事件条件至少选择事件编码或事件类型之一。
- 状态条件中持续时长非必填；若填写，需大于 0。
- 时间条件非必填，但若配置则必须完整。

### 9.4 动作节点配置
功能目标：定义规则命中后要执行的动作。

#### 9.4.1 支持的动作类型
- 发送站内通知
- 发送告警消息
- 记录执行日志
- 设备打标签
- 设备移除标签
- 调整设备分组
- 调用设备服务

#### 9.4.2 发送站内通知
用于通知平台内运营或运维人员。

配置项：
- 通知标题
- 通知内容模板
- 接收对象范围

规则：
- 标题必填
- 内容支持引用设备名称、设备编码、属性值、事件内容等变量

#### 9.4.3 发送告警消息
用于触发告警中心或消息中心能力。

配置项：
- 告警级别
- 告警标题
- 告警内容
- 接收渠道

规则：
- 本期渠道可仅定义为平台内消息或预留扩展字段
- 告警内容支持变量占位

#### 9.4.4 记录执行日志
用于将规则命中行为沉淀到日志。

配置项：
- 日志级别
- 日志内容模板

#### 9.4.5 设备打标签
用于为命中规则的设备自动绑定指定标签。

配置项：
- 目标标签

规则：
- 若设备已绑定该标签，则本次执行视为幂等成功。

#### 9.4.6 设备移除标签
用于为命中规则的设备自动移除指定标签。

配置项：
- 目标标签

规则：
- 若设备当前未绑定该标签，则本次执行视为幂等成功。

#### 9.4.7 调整设备分组
用于将设备归入新的目标分组。

配置项：
- 目标分组

规则：
- 若目标分组处于禁用状态，则动作配置不可保存或执行时报错。

#### 9.4.8 调用设备服务
用于触发产品定义的设备服务。

配置项：
- 服务标识
- 服务参数模板
- 超时时间

规则：
- 仅允许选择所属产品已定义的服务。
- 参数模板需校验基础格式。
- 若设备处于停用或离线状态，可根据业务规则限制执行。

#### 9.4.9 动作执行规则
- 支持多个动作节点顺序执行。
- 动作节点按配置顺序执行。
- 动作失败策略支持：
  - 继续执行后续动作
  - 中断执行
- 若未配置动作节点，则规则不可保存。

### 9.5 规则测试
功能目标：帮助用户在正式启用前验证规则逻辑。

#### 9.5.1 支持能力
- 使用指定设备进行规则测试
- 使用模拟输入进行规则测试
- 返回条件命中结果
- 返回动作预执行结果

#### 9.5.2 测试输出
- 是否命中条件
- 命中的是哪些条件节点
- 将执行哪些动作节点
- 哪些动作节点可能失败及原因

### 9.6 规则执行日志
功能目标：帮助用户查看规则执行历史与执行结果。

#### 9.6.1 展示字段
- 规则名称
- 触发时间
- 触发设备
- 条件命中结果
- 动作执行结果
- 整体执行状态
- 失败原因

#### 9.6.2 支持能力
- 按规则筛选
- 按设备筛选
- 按执行状态筛选
- 按时间范围筛选
- 查看单次执行详情

## 10. 条件节点设计原则
- 条件节点用于表达“判断逻辑”，不承载执行动作。
- 条件配置应尽量结构化，避免自由脚本。
- 条件节点要与设备属性、设备事件、设备状态的实际数据模型保持一致。
- 条件节点需支持清晰的可读表达，例如“温度 > 40 且持续 5 分钟”。

## 11. 动作节点设计原则
- 动作节点用于表达“执行逻辑”，不承载条件判断。
- 动作节点应以幂等为优先原则，避免重复执行造成异常结果。
- 动作节点需支持失败回执和执行日志记录。
- 本期动作节点按顺序串行执行，不支持并行分支。

## 12. 状态设计

### 12.1 规则状态
- 草稿
- 启用
- 停用

### 12.2 状态流转规则
- 草稿规则允许编辑、测试、删除。
- 启用规则允许停用，不建议直接删除。
- 停用规则允许编辑、再次启用或删除。

### 12.3 执行状态
- 未触发
- 已触发
- 执行成功
- 部分成功
- 执行失败

## 14. 业务流程

### 14.1 创建规则流程
1. 进入规则新建页。
2. 填写规则基础信息。
3. 选择适用范围。
4. 配置条件节点。
5. 配置动作节点。
6. 测试规则。
7. 保存草稿或直接启用。

### 14.2 规则执行流程
1. 设备数据上报或状态变化触发规则扫描。
2. 规则引擎判断适用范围。
3. 规则引擎计算条件节点是否命中。
4. 命中后按顺序执行动作节点。
5. 记录执行日志。

### 14.3 异常处理流程
1. 规则执行失败。
2. 系统记录失败动作节点和失败原因。
3. 用户在执行日志页查看失败明细。
4. 用户修改规则并重新测试。

## 15. 接口建议
- POST /rule/page：规则分页查询
- GET /rule/detail：规则详情查询
- POST /rule/create：创建规则
- POST /rule/update：更新规则
- POST /rule/enable：启用规则
- POST /rule/disable：停用规则
- POST /rule/delete：删除规则
- POST /rule/test：测试规则
- POST /rule/log/page：规则执行日志分页查询
- GET /rule/log/detail：规则执行日志详情查询

## 16. 非功能要求
- 规则编码需全局唯一。
- 规则保存时需完成基础字段校验、条件节点校验和动作节点校验。
- 规则执行需保留执行日志。
- 条件判断和动作执行失败时需返回明确原因。
- 规则列表与日志列表需支持常规查询性能要求。
- 动作节点设计需具备基础幂等能力。

## 17. 验收标准

### 17.1 功能验收
- 可创建、编辑、启用、停用、删除规则。
- 可配置至少一种条件节点和至少一种动作节点。
- 可通过测试能力验证规则配置结果。
- 可查看规则执行日志及失败原因。

### 17.2 规则能力验收
- 支持基于设备属性、事件、状态配置条件节点。
- 支持发送通知、记录日志、标签操作、分组调整、设备服务调用等动作节点。
- 支持条件节点组合关系配置。
- 支持动作节点顺序执行。

### 17.3 交付验收
- 产品、设计、研发、测试可基于本 PRD 继续拆解详细方案。
- 页面、接口、字段、规则和验收标准之间无明显冲突。
